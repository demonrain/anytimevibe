# 随码多用户容量评估

## 结论

当前 Relay 不运行 Codex，也不读取代码正文，只负责认证、WebSocket 路由、密文事件存储和 Web Push。一般使用强度下，`2 vCPU / 4 GB RAM / 40 GB SSD` 可作为约 100 个注册用户、20 个同时在线用户的起点；约 500 个注册用户、100 个同时在线用户建议使用 `4 vCPU / 8 GB RAM / 100 GB SSD`，并将 PostgreSQL 独立部署或使用托管数据库。

这些数字是容量规划起点，不是性能承诺。真实压力主要由同时在线设备数、任务完成快照大小、历史保留时长和登录峰值决定。

## 压力来源

| 资源 | 主要来源 | 估算方式 |
| --- | --- | --- |
| CPU | Argon2 密码校验、TLS、JSON 校验、Web Push | 登录高峰通常比消息转发更消耗 CPU |
| 内存 | 浏览器与 Agent WebSocket、消息缓冲、Node.js 基础开销 | 每条连接按约 50-150 KB 应用层开销预留，并保留突发缓冲空间 |
| 数据库 | 用户、会话、主机和端到端加密事件 | 密文事件通常是主要增长项 |
| 带宽 | 任务状态、线程同步快照和审批事件 | 与活跃任务数和同步频率近似线性增长 |

## 参考场景

假设每个在线用户平均连接一个浏览器和一个 Agent，每条密文事件平均 4-16 KB，每个用户每月产生 2,000 条持久化事件：

- 100 个注册用户的月新增事件数据约为 0.8-3.2 GB，索引和 PostgreSQL 膨胀建议额外预留 30%-60%。
- 20 个同时在线用户约有 40 条长期 WebSocket，连接本身压力很小，集中同步线程快照才是主要内存变量。
- 若每个活跃用户平均传输 100 KB/分钟，20 个用户持续活跃一天约产生 2.9 GB 单向业务流量。实际使用通常不是全天持续活跃。
- Argon2 校验是有意设计的高成本操作。开放注册后应保留当前登录和注册限流，避免 CPU 被集中认证请求占满。

## 部署建议

| 规模 | 建议配置 | 数据库建议 |
| --- | --- | --- |
| 1-100 注册用户，20 同时在线 | 2 vCPU / 4 GB / 40 GB SSD | Compose 内 PostgreSQL 可用，每日备份 |
| 100-500 注册用户，100 同时在线 | 4 vCPU / 8 GB / 100 GB SSD | 独立或托管 PostgreSQL，启用连接和慢查询监控 |
| 500+ 注册用户或 200+ 同时在线 | 8 vCPU 起，Relay 水平扩容 | 托管 PostgreSQL；WebSocket 需要共享在线路由或粘性会话 |

当前在线 Agent 和浏览器连接保存在单个 Relay 进程内。水平扩容前必须增加 Redis/NATS 一类的跨实例路由与在线状态共享，不能直接启动多个无状态 Relay 实例。

## 运维阈值

- Relay 持续 CPU 超过 70% 或事件循环延迟持续超过 100 ms 时扩容或检查认证攻击。
- 内存持续超过限制的 75% 时检查 WebSocket 数、消息突发和 2 MiB 最大消息配置。
- PostgreSQL 磁盘达到 70% 时扩容磁盘或制定密文事件保留策略。
- 数据库查询 P95 超过 100 ms 时检查 `sync_events(host_id, sequence)`、用户主机查询和数据库 IOPS。
- 监控注册失败、登录失败、WebSocket 断连率、在线 Agent 数、每分钟持久化事件数和 Push 失败率。

## 多用户安全边界

- 用户、主机、同步事件和 Push 订阅均按 `user_id` 校验。
- 注册默认开启，可通过 `REGISTRATION_ENABLED=false` 关闭。
- `MAX_USERS` 限制总注册人数，默认 100。
- 用户名不区分大小写且保持唯一。
- 当前不包含邮箱验证、密码找回、管理员封禁、配额计费和审计后台。公开服务前建议至少补充验证码或邀请制、密码重置和滥用处置流程。
